## 项目定位

ACE / Anchor Capability Engineering 是我用于组织 AI Agent 参与复杂任务的一套工程协作方法论。

它不是一个单独工具，不是某个 Agent 的使用技巧，也不是企业级平台。它更像一套围绕真实项目逐步形成的工作方式：如何拆任务、分角色、控风险、留证据、做验收、沉淀 Skill，并让 ChatGPT、Codex、Claude Code、WorkBuddy 等不同工具在同一个工程链路里发挥作用。

ACE 的目标不是制造一个复杂系统，而是建立一套普通人也能使用的轻量工程协作方法，让 AI 从“聊天助手”进一步变成可以参与长期项目建设的工程协作者。

## 为什么做 ACE

在实际使用 AI 的过程中，我发现真正困难的不是让 AI 回答一个问题，而是让 AI 稳定参与一个复杂项目。

一个真实项目通常不是一次对话能完成的。它可能涉及需求整理、方案设计、代码修改、环境配置、版本管理、测试验收、文档沉淀和后续迭代。如果没有一套方法，AI 很容易出现几个问题：

- 任务边界不清楚。
- 改动过程不可追踪。
- 环境和依赖被反复误判。
- 一个 Agent 从头到尾自说自话，没人换角度审查。
- 验收要么过度复杂，要么完全缺失。
- 有效经验没有沉淀，下次又重新踩坑。

ACE 就是为了解决这些问题而形成的。它关注的不是“怎么写一个更厉害的提示词”，而是“怎么组织 AI 参与真实工程任务”。

## 它解决什么问题

ACE 主要解决四类问题。

第一，任务怎么拆。不是所有任务都需要复杂流程，但涉及多文件、多模块、环境依赖、生产链路或不可逆操作时，就不能让 AI 直接开干。

第二，Agent 怎么分工。Codex、Claude Code、WorkBuddy、ChatGPT 各有优势，真正的问题不是哪个最强，而是如何让它们在同一套任务系统里分工、交接、审查和复用成果。

第三，工程事实怎么保留。AI 项目最怕没有事实层。如果没有明确的环境事实源、版本事实层、验收证据和回滚机制，项目越做越乱，AI 越改越危险。

第四，方法怎么复用。有效经验如果只留在聊天记录里，几乎等于没有沉淀。ACE 会把反复有效、高风险、容易失败、能提高效率的做法沉淀成 Skill。

## 核心方法

ACE 当前由几个互相连接的部分组成：任务分层、角色动态分配、Agent 协作、Git / Jujutsu 版本事实层、环境事实源、最少充分验收和 Skill 沉淀。

这些部分不是为了搭一个大系统，而是为了让复杂项目在 AI 参与后仍然保持边界、事实、验证和复用。

## 任务分层

ACE 会根据任务风险和复杂度决定处理方式。

- 简单任务：直接完成，不增加流程负担。
- 中等任务：拆分步骤，保留关键依据。
- 高风险任务：先做蓝图，再施工，再验收。
- 多 Agent 任务：明确角色、边界、交接和审查方式。

核心原则是：流程为任务服务，不为了流程而流程。轻任务不加流程，复杂任务才进入工程化路径。

## 角色动态分配

ACE 不把某个模型固定绑定到某个角色。

不同任务需要不同能力。有些 Agent 适合做架构，有些适合写代码，有些适合审查，有些适合总结和沉淀。因此，ACE 强调根据任务需要动态分配角色，例如：

- 总工 / 架构 Agent。
- 施工 Agent。
- 审查 Agent。
- 测试 Agent。
- 文档 / 复盘 Agent。
- Skill 沉淀 Agent。

这套分工不是为了模拟公司组织架构，而是为了避免一个 Agent 从头到尾闭门造车，让复杂任务里始终有分工、交接和检查。

## Agent 协作

ACE 支持多个 AI 工具在同一项目里协作。

在我的工作流里，ChatGPT 更常承担总工判断、方案设计、复盘总结和方法论整理；Codex 更常承担代码实现、局部修改和工程施工；Claude Code 适合复杂工程任务、代码审查或独立验证；WorkBuddy 更贴近 Windows 本地执行、日志整理、路径检查和实际操作辅助。

工具不是重点，协作方式才是重点。ACE 真正关心的是：如何让不同 Agent 围绕同一个任务事实协同工作，而不是各说各话。

## Git / Jujutsu 版本事实层

ACE 采用 Jujutsu + Git 的双层路线。

Jujutsu 更适合作为本地施工、变更整理、撤销和 workspace 管理前端。Git 保留为兼容、交换、commit SHA、diff、CI、审计和协作账本的事实层。

这套组合的价值在于：让 AI 的工程改动有明确记录，方便审查、回滚和交接。在多 Agent 协作中，版本事实层尤其重要。不同 Agent 应该围绕真实 diff 和 commit 讨论，而不是只依赖聊天记录。

## 环境事实源

ACE 很重视环境事实源。

在 AI 工程协作中，很多问题不是因为模型不聪明，而是因为环境事实混乱。例如：到底在哪个目录工作，使用哪个 Python / Node / 运行时，正式入口文件是什么，依赖锁定文件在哪里，哪些配置是当前生效的，哪条命令才是可信启动命令。

ACE 的原则是：找不到，不等于没安装；报错，不等于应该重装；Agent 不应该无边界搜索环境。

每个真实项目都应该有一个轻量但明确的环境事实源，减少重复排查和错误修复。

## 最少充分验收

ACE 不追求每次都做完整大验收。

它强调只验证本次改动直接影响的关键链路。小改动只做局部验证；配置改动验证配置是否被正确读取；环境改动验证正式运行路径；生产链路改动才做完整回归；高风险任务才保留完整证据包。

这可以避免两种极端：一种是完全不验收，导致项目失控；另一种是为了显得严谨而过度验收，反而拖慢真实推进。

## Skill 沉淀

ACE 的长期复利来自 Skill。

但 Skill 不是随便写笔记，也不是把所有过程都文档化。ACE 的原则是：先做事，后沉淀。只有真正可复用、高风险、反复失败、能提升效率的能力，才值得沉淀成 Skill。

ACE-Skill-Foundry 的定位不是大而全的知识库，而是一个轻量能力沉淀层。它记录真实项目中反复有效的做法，让后续项目可以直接复用。

## 当前状态

ACE 目前不是成熟商业产品，也不是完整自动化平台。

它更准确的状态是：一套正在真实项目中持续进化的个人 AI 工程协作方法论。当前已经形成了 Jujutsu + Git 后端路线、轻量工程规则、环境事实源原则、最少充分验收原则和 Skill 沉淀机制，并在 Anchor AI Lab、TemuCanvas、AI PPT Workflow 等项目中持续验证。

## 和其他项目的关系

TemuCanvas 是 ACE 在图像处理和商品图自动化场景里的实践样本，用于组织数据分层、候选生成、人工复核、错误放行控制、失败诊断和版本冻结。

AI PPT Workflow 是 ACE 在内容生产和演示文稿制作场景里的实践样本，用于组织任务预检、内容战略、逐页计划、静态版验收和动效增量施工。

AI Tool Reviews 使用 ACE 的工作流视角评估 AI 工具、开源项目和生产力方案是否值得进入真实工作流。

Anchor AI Lab 是所有项目的公开出口，ACE 是其中最核心的方法论资产之一。

## 后续方向

下一步，ACE 会继续沿着真实项目往前沉淀，而不是先做成复杂平台。

重点方向包括：继续整理公开项目页和文章；把稳定做法沉淀为更清晰的 Skill；在 TemuCanvas、AI PPT Workflow、AI Tool Reviews 等项目中持续验证；保留轻量原则，避免为了完整而完整。
